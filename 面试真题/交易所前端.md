## 1. 三明治攻击如何防范

**场景：** USDT 买 ETH，金主 10wU 买 ETH

针对大额交易，通过提高 gas 费的手段，抢先打包高价的交易

### 介绍什么是三明治攻击？

**前置交易：** 目的在于抬高 AMM 价格

**\_；** 用户（金主）的大额交易

**后置交易：** 目的在于卖出前面买入的代币，赚取中间的价格差

### 如何防范？

1. **交易滑点**（失败了 gas 费还是丢了）
2. **系统的风控**
3. **攻击者知道有这么一比大额交易**

首先我们在交易所的交易，前端会提交到 rpc 节点，这里攻击者往往是节点的控制者（能够读写里面的内容）

**Flashbots Protect RPC**

**用户：** 前端连接交易所私有的 rpc（协议，交易的签名算法可能会微调一些逻辑）

## 2. 中心化交易的 nonce 能否不连续

首先 nonce 其实是一个索引号，一般是开放给中下游的公司用来查询交易用的

这里用户下单，理想情况是陆续 1，2，3

实际上很多是并行的；1，2，3 => 2，1，3 => 3,2,1

**序号设置连续的目的：** 就是要知道下单的顺序 => 单调性（递增），保证唯一性

**场景：** 高并发的时候，如果是连续的话，（没处理好锁）可能会出现多个用户下的单 nonce 是一样的

**解决方案：** 雪花算法 snowflare

## 3. 有没有了解过零知识证明？

### 三个角色

**证明者、求证者、旁观者**

**要求：** 证明者和求证者（可以证明这个问题）的对话是公开透明的；但是旁观者所看到的、听到的没法证明这个问题

### 举例说明

张三要想李四证明，自己是李四的父亲；王五在一边吃瓜；

张三和李四说，我知道你身上哪个地方有块 zhi；（李四身上有没有 zhi，只有李四知道，王五并不知道）

李四说，如果你是我的父亲，请你帮我买辆车；张三毫不犹豫，就给李四转了 100w。（李四知道自己和张三之前没什么交情往来，张三这么大方，那应该是亲爹了；但是王五看来，可能觉得他两只是前面有生意、交情上的往来，没法证明亲子关系）

### 核心

求证者和证明者之间通过信息差来证明，让旁观者无法得到有价值的信息

### 用途

- Layer2 扩容（Layer1 存储成本高，Layer2 便宜，但是 Layer1 凭什么信任 Layer2 的数据？）
- DAO 治理
- 投票

## 4. 有没有了解过闪电贷、杠杆？

### 套利场景

手上有 ETH、USDT、BTC、DAI

发现隔壁 DEX 有一个 BTC 对 DAI 的套利机会（BTC 对 DAI 的价格比市场价格高了，同样 1 个 BTC 能够在这个 DEX 换到更多的 DAI，再按照市场价，到正常的交易所给 DAI 再换成 BTC，实现套利）

咱们一把梭哈：

1. 借贷池，给手上的 ETH、USDT 抵押借出 BTC（可能得等 2-6 个区块，20-60s）
2. 将所有的 BTC 在隔壁 DEX 换成 DAI
3. 跑到正常价格交易所，换成 BTC
4. BTC 支付利息并赎回 ETH、USDT

**问题：** 套利机会可能只有几秒钟的时间，犹豫就会白给

### 闪电贷的核心

给上面 1，2，3，4 操作都在一个区块的一比交易里面完成

### 智能合约怎么知道我这个套利操作具体是啥？

```solidity
function flashSloan(address target, bytes calldata data) {
  // ...上面资金池借贷的逻辑
  // 我只要知道这个套利的合约地址（隔壁 DEX 的 Router 合约），还有相关的函数就可以了
  target.call(data){value: msg.value};
  // ...
}
```

**data 类型：** bytes（前 4 个字节是啥？）

函数选择器 selector 长度 4 个 byte；光知道函数选择器，是不是还需要知道函数的入参

```solidity
exchange(uint amount0In, uint amount1In);
abi.encode(selector, uint amount0In, uint amount1In) => bytes
```

### 杠杆

按照上面套利的场景，用户的需求是尽可能多借到一些 BTC，可以套更多

我们还可以继续抵押借到的 BTC，去借更多的 BTC（利息，得在一个区块里面归还）

**本质：** 以贷养贷

**后端：** 有一个健康因子（避免超出安全范围）

## 6. 怎么知道用户的网站卡顿了？

**目的：** 知道用户浏览哪个页面（history），发生了卡顿，上报到公司的服务器

### 方案一：使用付费 Sentry 性能监控插件

### 方案二：自己实现（为了节约成本）

**WebWorker：**

**栈、微队列、宏队列的执行顺序？**

先清空栈，栈清空了再清空微队列，最后有时间执行宏队列（但是不清空）

```javascript
// 网站里面设置一个定时器，每隔 1s 传递一次时间给到 webWorker
setInterval(() => {
  window.postMessage(performance.now());
}, 1000);

// webWorker 注册接收函数
let lastRepoTime;
window.onmessage = function (data) {
  // data 是 web 应用传递的时间戳
  if (!lastRepoTime) {
    lastRepoTime = data;
    return;
  }
  if (data - lastRepoTime > 1000 + 300) {
    // 发生了卡顿
    // 提交请求到服务器
    fetch("服务器地址", "当前的 url，用户信息，时间...");
  }
};
```

**原理说明：**

浏览器的 JS 解释进程和渲染进程是同步的（大部分情况是 JS 解释进程占用了过多的时间，导致渲染进程无法正常运行，让用户感到视觉上的卡顿）

react16 如果页面的计算逻辑复杂了，就会卡顿
