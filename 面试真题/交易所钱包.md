## Tree Shaking

### 1. 问：有哪些代码是可以进行 tree shaking 的，哪些是不可以的呢？

**答：**

可以 Tree Shaking 的代码：

```javascript
// ES6 模块的命名导出
export const add = (a, b) => a + b;
export const multiply = (a, b) => a * b;

// 只导入使用的函数
import { add } from "./math"; // multiply 会被删除
```

不能 Tree Shaking 的代码：

```javascript
// CommonJS 模块
module.exports = { add, multiply }; // 整个对象都会被包含

// 有副作用的代码
console.log("This has side effects"); // 会被保留

// 动态导入
const moduleName = "math";
import(moduleName); // 无法静态分析
```

### 2. 问：比如说我们在 JS 中有很多函数，什么样的函数会被摇掉不会被编译进来？什么样的代码会被编译进来呢？

**答：**

会被 Tree Shaking 的函数：

- 未被引用的导出函数
- 纯函数且无副作用
- 静态可分析的代码路径

不会被 Tree Shaking 的函数：

- 有副作用的函数（修改全局变量、DOM 操作等）
- 被动态引用的函数
- 构造函数和类方法（通常）

### 3. 问：假如说 class 里面有一些方法，那么这些方法其实我们并没有用到，那他会不会编译进来呢？

**答：** 通常会编译进来，原因：

```javascript
class MyClass {
  usedMethod() {
    return "used";
  }
  unusedMethod() {
    return "unused";
  } // 仍会被包含
}

const instance = new MyClass();
instance.usedMethod();
```

**原因分析：**

- JavaScript 的动态特性使得静态分析困难
- 方法可能通过字符串动态调用：`instance['unusedMethod']()`
- 继承和多态增加了复杂性
- 反射和 `Object.getOwnPropertyNames` 可能访问所有方法

**优化建议：**

```javascript
// 使用函数而不是类方法
export const usedFunction = () => "used";
export const unusedFunction = () => "unused"; // 可以被 tree shake

// 或使用/*#__PURE__*/注释
/*#__PURE__*/ class MyClass {
  // 告诉 bundler 这个类是纯的
}
```

## TypeScript

### 1. 问：你是怎么推进 TS 的重构的，还有你们的类型系统这些东西能展开聊一聊吗？

**答：** TS 重构策略：

**1. 渐进式迁移：**

- 配置 `allowJs: true`，允许 JS 和 TS 共存
- 使用 `@ts-check` 为 JS 文件添加类型检查
- 逐个文件从 `.js` 重命名为 `.ts`

**2. 类型定义：**

- 为第三方库安装 `@types/` 包
- 为内部模块编写 `.d.ts` 声明文件

**实践建议：**

- 一般当你接手到这个项目的时候，往往是一个逻辑比较杂乱的单页应用
- 先从 api 和 contract 远程调用（一般通过脚本、AI 工具生成 ts 类型）返回值入手（制定开发流程，得强制要求）
- 然后再去修改对应的业务组件（pages）

### 2. 问：TS 的高级类型能给我举一些例子吗？

};

### 3. 问：比如说我怎么取一个函数的返回值类型呢？还有这个函数的参数类型？

**答：**

```typescript
// 获取函数返回值类型
type ReturnType<T extends (...args: any) => any> = T extends (
  ...args: any
) => infer R
  ? R
  : any;

// 获取函数参数类型
type Parameters<T extends (...args: any) => any> = T extends (
  ...args: infer P
) => any
  ? P
  : never;

// 使用示例
function getUserInfo(id: number, name: string): { id: number; name: string } {
  return { id, name };
}

type UserInfo = ReturnType<typeof getUserInfo>; // {id: number, name: string}
type UserParams = Parameters<typeof getUserInfo>; // [number, string]
```

### 4. 问：这种能怎么样去拿到它的参数类型和返回值类型呢？

**答：**

```typescript
// 实际使用场景
function fetchUser(
  userId: number,
  options?: { cache: boolean }
): Promise<User> {
  // implementation
}

// 获取参数类型
type FetchUserParams = Parameters<typeof fetchUser>;
// [number, {cache: boolean}?]

// 获取返回值类型
type FetchUserReturn = ReturnType<typeof fetchUser>;
// Promise<User>

// 创建参数对象
const params: FetchUserParams = [123, { cache: true }];

// 创建高阶函数，保持类型安全
function withLogging<T extends (...args: any[]) => any>(fn: T) {
  return (...args: Parameters<T>): ReturnType<T> => {
    console.log("Calling function with args:", args);
    return fn(...args);
  };
}
```

```javascript
// 只渲染可见区域的列表项
const VirtualList = ({ items, itemHeight, containerHeight }) => {
  const [scrollTop, setScrollTop] = useState(0);

  const visibleStart = Math.floor(scrollTop / itemHeight);
  const visibleEnd = Math.min(
    visibleStart + Math.ceil(containerHeight / itemHeight),
    items.length
  );

  const visibleItems = items.slice(visibleStart, visibleEnd);

  return (
    <div
      style={{ height: containerHeight, overflow: "auto" }}
      onScroll={(e) => setScrollTop(e.target.scrollTop)}
    >
      <div style={{ height: items.length * itemHeight }}>
        <div
          style={{ transform: `translateY(${visibleStart * itemHeight}px)` }}
        >
          {visibleItems.map((item, index) => (
            <div key={visibleStart + index} style={{ height: itemHeight }}>
              {item.content}
            </div>
          ))}
        </div>
      </div>
    </div>
  );
};
```

## React 架构相关

### Fiber 架构和调度器

### 1. 问：介绍一下 fiber 架构这个东西，以及 React 调度器他们是怎么运行的机制？

**答：** Fiber 架构是 React 16 引入的新的协调引擎，主要解决了以下问题：

- 可中断渲染：将渲染工作拆分成小的工作单元，可以被中断和恢复
- 优先级调度：不同类型的更新有不同优先级（同步、异步、低优先级等）
- 数据结构：使用链表结构（child、sibling、return 指针）替代递归遍历

**调度器工作机制：**

- 使用 MessageChannel 和 requestAnimationFrame 实现时间切片
- 每个时间片（5ms）执行一部分工作，剩余时间交还给浏览器
- 根据任务优先级决定哪些任务先执行
- 高优先级任务（用户输入、动画）会中断低优先级任务（数据获取、列表渲染）

### 2. 问：调度器是在哪个库中？以及整套 fiber 算法是在哪个库中做的？

**答：**

- 调度器（Scheduler）：在独立的 `scheduler` 包中，也被称为 React Scheduler（通过最小堆的数据结构，结合任务的类型、剩余时间，做一个排序，给 `performUintOfWork` 函数一个合理的执行顺序）
- Fiber 架构：主要在 `react-reconciler` 包中实现
- ReactDOM：负责具体的 DOM 操作和平台相关的渲染逻辑
- React 核心包：只负责定义组件 API 和基础数据结构

### React 库关系

### 1. 问：React 和 ReactDOM 有什么关系？

**答：**

- React：核心库，负责组件定义、状态管理、生命周期等平台无关的逻辑
- ReactDOM：渲染器，负责将 React 组件渲染到 DOM 环境
- 架构设计：React 采用分层架构，核心与渲染分离
- 其他渲染器：React Native（移动端）、React Test Renderer（测试）等

### 2. 问：有了解过 React Native 吗？

**答：** React Native 是 React 的移动端渲染器：

- 跨平台：使用 JavaScript 开发 iOS 和 Android 应用
- 原生渲染：不是 WebView，而是调用原生组件
- Bridge 机制：JavaScript 与原生代码通过 Bridge 通信
- 热更新：支持代码推送更新，无需发布应用商店

### React 渲染机制

### 1. 问：假如说我们现在有一个组件，组件中有一个 state，还渲染了一个按钮。点按钮的时候会触发一个函数，在函数中会执行 setState，然后在 setState 之后会紧跟着一个 setTimeout，然后在 setTimeout 中又进行了 setState。那么我们的这个组件会进行几次渲染呢？

**答：**

- **React 17 及之前：** 3 次渲染

  - 第一次：同步的 setState
  - 第二次：setTimeout 中的 setState（异步）
  - 第三次：如果还有其他 setState

- **React 18：** 2 次渲染
  - 第一次：同步代码中的 setState 会被批处理
  - 第二次：setTimeout 中的 setState（现在也支持自动批处理）

### 2. 问：React 会有什么优化吗？为什么会有两次渲染呢？

**答：**

- 批处理优化：React 会将多个 setState 调用合并成一次渲染
- 时间切片：setTimeout 创建了新的执行上下文，脱离了当前批处理
- React 18 改进：通过 `createRoot` 启用自动批处理，即使在 Promise、setTimeout 中也会批处理

### 3. 问：如果我们在两次 setState 里传的值都是一个函数，而且 setTimeout 的第二个值是 0，那么 React 会渲染几次？

**答：**

- 仍然是 2 次渲染（React 18 中）
- 函数式更新确保状态基于最新值更新，但不影响批处理逻辑
- `setTimeout(fn, 0)` 仍然是异步执行，会在下一个事件循环中执行

### 区块链签名机制

### 1. 问：不同的链的签名差异上面都有什么？比如说以太坊他是怎么进行签名的？

**答：** 以太坊签名：

- 算法：ECDSA（椭圆曲线数字签名）
- 曲线：secp256k1
- 格式：v, r, s 三个参数
- 实现：`web3.eth.personal.sign()` 或 `ethers.Wallet.signMessage()`

**不同链的差异：**

- Bitcoin：使用相同的 secp256k1，但地址格式不同
- Solana：使用 Ed25519 算法，更高效
- Cosmos：支持多种签名算法
- Polkadot：使用 Sr25519 算法

### 2. 问：假如说给你一个私钥，你怎么通过这个私钥去做？或者说给你一个助记词，你怎么通过助记词去生成私钥？

**答：**

```javascript
// 通过私钥创建钱包
import { ethers } from "ethers";
const wallet = new ethers.Wallet(privateKey);

// 通过助记词生成私钥
const mnemonic = "word1 word2 ... word12";
const hdNode = ethers.utils.HDNode.fromMnemonic(mnemonic);
const wallet = new ethers.Wallet(hdNode.privateKey);
```

### 3. 问：怎么把助记词变成私钥？

**答：** 标准流程：

1. BIP39：助记词 → 种子（通过 PBKDF2）
2. BIP32：种子 → 主私钥（通过 HMAC-SHA512）
3. BIP44：主私钥 → 派生私钥（通过派生路径）

### 4. 问：一组助记词可以生成多少个私钥？

**答：**

- 理论上无限个：通过不同的派生路径
- 常用路径：`m/44'/60'/0'/0/0`, `m/44'/60'/0'/0/1` ...
- 路径格式：`m/purpose'/coin_type'/account'/change/address_index`
- 实际应用：钱包通常生成几十到几百个地址

### 5. 问：生成很多个私钥的话，那这些私钥都要保存吗？它为什么可以生成多个私钥？

**答：** 不需要保存所有私钥：

- 只需保存助记词和派生路径
- 可以根据需要重新生成任意私钥
- 确定性生成：相同助记词 + 路径 = 相同私钥

**生成原理：**

- 基于数学上的单向函数
- 使用 HMAC-SHA512 进行密钥派生
- 每个路径对应唯一的私钥

### 钱包插件开发

### 1. 问：做过钱包插件，一个 web 网页怎么样去调取你的钱包，然后怎么样去给你做交互，能拿到签名，能给我讲一讲这个运行机制吗？

**答：** 交互流程：

1. 注入 Provider：插件向页面注入 `window.ethereum` 对象
2. 连接请求：网页调用 `ethereum.request({method: 'eth_requestAccounts'})`
3. 用户授权：插件弹出授权弹窗，用户确认
4. 返回账户：插件返回用户授权的账户地址
5. 签名请求：网页发送签名请求
6. 用户确认：插件弹出签名确认弹窗
7. 返回签名：用户确认后返回签名结果

**技术实现：**

- EIP-1193：以太坊 Provider 标准
- JSON-RPC：使用 RPC 协议通信
- 事件系统：支持账户变更、网络切换等事件

### 2. 问：假如说我现在需要调取你的签名，你会弹出你插件的哪一块内容呢？

**答：** 会弹出签名确认弹窗，包含：

- 交易详情：from、to、value、data、gas 等
- 费用估算：gas fee、总费用
- 风险提示：合约交互警告、高费用警告等
- 确认按钮：确认/拒绝操作

### 3. 问：钱包插件有哪些部分组成？比如说 background.js、content、popup 他们都是怎么样协调三者之间的通信的？

**答：** 插件组成：

- Background Script：后台服务，处理核心逻辑
- Content Script：注入到网页，提供 API 接口
- Popup：用户界面，显示账户信息和交易确认
- Options Page：设置页面，管理账户和网络

**通信机制：**

```
网页 ←→ Content Script ←→ Background Script ←→ Popup
(postMessage) (chrome.runtime) (chrome.runtime)
```

### 4. 问：通信这一块，你用是直接用的浏览器的 API，还是说自己从底层封装了一套来做的？

**答：** 混合方式：

- 底层：使用浏览器原生 API (`chrome.runtime.sendMessage`)
- 封装层：创建事件总线和消息路由系统
- 类型安全：TypeScript 定义消息类型
- 错误处理：统一的错误处理和重试机制

### 5. 问：整个插件架构是由别人去做的吗？你对整体的架构其实并不是特别清晰是吗？

**答：** 架构分层：

- UI 层：React + TypeScript 构建的用户界面
- 状态管理：Redux/Zustand 管理应用状态
- 业务逻辑层：账户管理、交易处理、网络切换
- 数据持久层：加密存储私钥和用户设置
- 网络层：与区块链节点通信

### 6. 问：插件的架构，比如说不同插件的不同 JS 它是怎么进行通信的，通讯机制是什么？比如说你怎么样让 background 去调用 popup 弹出？

**答：** 通信流程：

```javascript
// Background 调用 Popup
chrome.action.openPopup(); // 打开 popup

// 发送消息到 popup
chrome.runtime.sendMessage({
  type: "TRANSACTION_REQUEST",
  data: transactionData,
});

// Popup 接收消息
chrome.runtime.onMessage.addListener((message, sender, sendResponse) => {
  if (message.type === "TRANSACTION_REQUEST") {
    // 显示交易确认界面
    showTransactionConfirm(message.data);
  }
});
```

**消息路由系统：**

- 事件总线：中央消息分发器
- 消息类型：定义标准消息格式
- 权限控制：验证消息来源和权限
- 异步处理：支持 Promise 和回调

### 模块联邦

### 1. 问：模块联邦能介绍一下吗？

**答：** **模块联邦（Module Federation）**是 Webpack 5 引入的微前端解决方案：

**核心概念：**

- Host 应用：消费远程模块的主应用
- Remote 应用：提供模块的远程应用
- Exposed 模块：远程应用对外暴露的模块
- Shared 依赖：应用间共享的依赖库

**配置示例：**

```javascript
// Remote 应用配置
new ModuleFederationPlugin({
  name: "remote_app",
  filename: "remoteEntry.js",
  exposes: {
    "./Button": "./src/Button.jsx",
  },
  shared: ["react", "react-dom"],
});

// Host 应用配置
new ModuleFederationPlugin({
  name: "host_app",
  remotes: {
    remoteApp: "remote_app@http://localhost:3001/remoteEntry.js",
  },
  shared: ["react", "react-dom"],
});
```

### 2. 问：他和乾坤有什么差异呢？

**答：** 主要差异：

- 版本管理：模块联邦支持版本控制和依赖解析
- 类型安全：支持 TypeScript 类型定义共享
- 构建优化：支持 tree shaking 和代码分割
- 热更新：支持开发时的热更新
- 依赖共享：自动处理共享依赖，避免重复加载
- 控制权：自主控制模块的暴露和消费

### 3. 问：我们可以用模块联邦来做一些，比如说除了远程模块加载以外，还能做些什么事情呢？使用场景能详细展开聊一聊吗？

**答：** 使用场景：

1. 组件库共享：设计系统组件在多个应用间共享
2. 业务模块复用：登录、支付等通用业务模块
3. 渐进式迁移：老项目逐步迁移到新技术栈
4. 团队协作：不同团队独立开发和部署
5. AB 测试：动态加载不同版本的功能模块
6. 插件系统：构建可扩展的插件架构

- 注意：不同工具间的互操作性需要特别处理

### 1. 问：假如说我们要使用 SSG，我们有大概 300 个页面。那么它每次编译的话可能要消耗很久的时间。那么有什么办法去优化吗？

### 2. 问：我们能不能让他不要在编译的时候就给他构建出来 HTML，反而是在第一次请求的时候去进行渲染，就是检测有没有缓存。如果有缓存的话就直接返回。如果没缓存的话，我们就给它渲染到 HTML 静态文件中。可以这样做吗？

**答：**

- 需求是懒加载，第一次请求该页面后，缓存计算结果
- Nextjs 的 middleware 中，next 程序缓存一个构建的 map `{'页面的路径'： '页面的 HTML 代码'}`
- 如果这个路径，没有在 map 里面读取到
- 执行页面构建出 html，然后 `map['页面的路径'] = '页面的 HTML 代码'`
