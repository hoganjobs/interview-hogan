## 2. 稳定性 & 监控

### Q2：你说的前端监控，具体监控了哪些内容？

**A2：**

- 性能指标监控：LCP、CLS、INP。
- 错误监控：资源错误、代码错误、Promise 错误、白屏检测。

### Q3：这些监控指标是怎么实现的？

**A3：**

- 重写浏览器原生事件（资源加载失败事件）。
- 监听 onerror 捕获错误。
- 使用订阅中心统一处理回调，格式化数据，上报到后端平台。

### Q4：你提到的 LCP，你是怎么定义的？

**A4：**

- 取页面最大内容元素的渲染时间作为 LCP。

### Q5：LCP 的统计范围，用户交互时怎么处理？

**A5：**

- 如果用户交互（点击、滚动），停止监听并上报。
- 如果没有交互，取页面最大内容渲染时间作为最终值。

## 3. 性能优化案例

### 3.1 SKU 状态优化

### Q6：你说到 INP 优化到 100ms 以下，具体是怎么做到的？

**A6：**

- **原来 SKU 逻辑维护 5 个状态：** 规格选择、可选状态、后端数据、Case、规格映射。
- **导致过多状态切换，** 交互时 INP 超过 100ms。
- **优化后：**
  - 仅保留「已选择规格」状态。
  - 其他不变数据用 `useMemo` 缓存。
  - 其他值按需计算。
- **优化结果：** INP 优化至 100ms 以下。

### Q7：之前 SKU 的状态为什么会多达 5 个？

**A7：**

- 历史实现方式过于复杂，尝试将所有可选信息都放到 state 中，导致重复维护。

### Q8：你是怎么把状态精简的？

**A8：**

- 保留必要的「已选择规格」。
- 其他用计算逻辑和缓存解决，避免冗余状态。

### Q9：INP 优化的结果，你是怎么验证的？

**A9：**

- 使用 Performance API 进行埋点上报。
- 验证点击交互时的输入延迟明显下降。

### 3.2 订单列表优化

### Q10：订单列表为什么会有性能问题？

**A10：**

- 大型组件耦合：状态管理、数据请求、渲染逻辑混杂。
- Tab 切换会重复请求。
- 所有 DOM 一次性渲染，仅通过 hidden 隐藏，浪费严重。

### Q11：你当时是怎么拆分订单列表组件的？

**A11：**

- 架构拆分：函数式组件 + hooks。
- 封装 `useOrders` 处理请求、缓存、分页，结合 Relay。
- 虚拟列表优化渲染。
- 将订单状态、按钮状态抽离为独立组件，结合 `React.memo`。

### Q12：你说自己写了虚拟列表，没有用第三方库，是出于什么考虑？

**A12：**

- 业务场景简单（订单固定高度）。
- 不需要复杂特性，所以自行实现轻量版虚拟列表。

### Q13：自己实现虚拟列表的关键点在哪里？

**A13：**

1. 计算可视区域的起止索引。
2. 使用 absolute + transform 定位，避免文档流重排。
3. 减少不必要的 DOM 渲染。

### Q14：你提到用 absolute + transform 定位，为什么不用 margin 或 position 相对定位？

**A14：**

- `absolute + transform` 可以减少文档流影响，避免触发回流重排，性能更好。

### Q15：你的虚拟列表支持"滚动到底加载更多"吗？

**A15：**

- 支持，通过监听滚动触发数据请求并更新渲染范围。

### Q16：如果要在前面插入新数据（比如消息流场景），你的虚拟列表能处理吗？

**A16：**

- 当前订单场景没有这种需求，所以未处理。
