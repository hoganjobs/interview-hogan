**什么是DApp，它与App有什么不同？有什么不同？**

答：App通常包含一个客户端，这个客户端会与一些中心化的资源（由一个组织拥有）进行通信，通常客户端通过一个中间层连接到中心化的数据层，如果中心化的数据层中的信息丢失，不能很轻松地恢复。
DApp表示去中心化应用程序。 DApps通过智能合约与区块链网络进行交互。 DApp使用的数据驻留在合约实例中。
中心化数据可能比去中心化数据更容易受到破坏。


**DApp的前端是否局限于某些技术或框架？**

答：不受限制。可以使用任何技术来开发DApp的前端，比如HTML，CSS，JS，Java，Python…

**前端用什么库连接后端（智能合同）？**

答：Web3.js库。

**在DApp的前端需要哪些东西才能与指定的智能合约进行交互？**
答：合约的ABI和字节码。

**ABI有什么作用？**
答：ABI是合约的公开接口描述对象，被DApps用于调用合约的接口。

**字节码有什么作用？**

答：节点上的EVM只能执行合约的字节码。

**为什么要使用BigNumber库？**
答：因为Javascript不能正确处理大数。

**为什么需要检查在Web DApp代码的开始部分是否设置了web3提供器（Provider）？**

答：因为Metamask会注入一个web3对象，它覆盖其他的web3设置。

**为什么要使用web3.js版本1.x而不是0.2x.x？**

答：主要是因为1.x的异步调用使用Promise而不是回调，Promise目前在javascript世界中
是处理异步调用的首选方案。

**如何在web3 1.x中列出账户？**

答：web3.eth.getAccounts

**.call和.send有什么区别？**

答：.send发送交易并支付费用，而.call查询合约状态。

**这样发送1个以太对吗：.send({value:1})？**

A：不对，这样发送的是1 wei。 交易中总是以wei为单位。

**那么为了发送1个以太，我必须将这个值乘以10^18，有更好的办法吗？**

答：有几种方法可以处理：

1. 使用 web3.utils.toWei() 方法（推荐）：
   
   ```javascript
   web3.utils.toWei('1', 'ether')    // 返回 '1000000000000000000'
   ```

2. 使用 BigNumber：
   
   ```javascript
   const BigNumber = require('bignumber.js');
   new BigNumber('1').multipliedBy(10 ** 18).toString()
   ```

3. 也可以使用 ethers.js 的工具方法：
   
   ```javascript
   ethers.utils.parseEther('1')    // 返回 BigNumber 对象
   ```

注意事项：

- 直接使用 JavaScript 的乘法（如 1 * 10^18）可能导致精度问题
- 在处理以太币金额时，始终使用字符串而不是数字，以避免精度损失
- wei 是以太坊中最小的计量单位，1 ETH = 10^18 wei

**调用.send()时需要指定什么？**

答：必须指定from字段，即发送账户地址。 其他一切都是可选的。

**web3.eth.sendTransaction()的唯一功能是将以太发送到特定的地址，这个说法是否正确？**

答：不对，也可以用它调用合约方法。

**你是否知道以太坊的可扩展性解决方案？**

答：以太坊的主要可扩展性解决方案包括：

1. 二层（Layer 2）扩展解决方案：
   
   - Rollups（卷叠）
     * Optimistic Rollups（乐观卷叠）：如 Optimism、Arbitrum
     * ZK Rollups（零知识卷叠）：如 zkSync、StarkNet
   - 状态通道（State Channels）：如 Lightning Network
   - Plasma：如 Polygon（原 Matic）
   - Sidechains（侧链）：如 xDai

2. 以太坊主网（Layer 1）优化：
   
   - 分片（Sharding）：将网络分割成多个并行处理的分片
   - EIP-1559：改善费用市场机制
   - PoS转换：从工作量证明转向权益证明，提高能效

这些解决方案各有优势：

- Rollups 提供了最好的安全性和去中心化平衡
- 状态通道适合频繁的小额支付
- Plasma 适合特定应用场景的扩展
- 分片将显著提升主网的交易处理能力

**DApp的前端开发有哪些特点？**

答：DApp前端开发具有以下特点：

1. 需要与区块链交互，通常通过Web3.js或Ethers.js等库
2. 需要处理钱包连接和用户授权
3. 需要考虑交易确认时间和用户体验
4. 需要处理智能合约的异步调用
5. 需要考虑gas费用和网络切换

**DApp前端开发常用的技术栈有哪些？**

答：常用的技术栈包括：

1. 框架：React、Vue、Angular等
2. Web3库：Web3.js、Ethers.js
3. 钱包连接：Web3Modal、WalletConnect
4. UI组件库：Material-UI、Ant Design
5. 状态管理：Redux、Vuex
6. 开发工具：Hardhat、Truffle

**在DApp前端开发中如何处理用户钱包连接？**

答：处理钱包连接通常需要：

1. 检测用户是否安装了钱包（如MetaMask）
2. 请求用户授权连接
3. 监听钱包账户变化
4. 处理网络切换
5. 保存用户的连接状态

**DApp前端如何优化用户体验？**

答：主要从以下几个方面：

1. 提供交易状态的实时反馈
2. 使用loading状态展示区块确认过程
3. 优化gas费用估算
4. 实现错误处理和友好提示
5. 提供交易历史记录
6. 支持多钱包集成

**如何在DApp前端处理智能合约事件？**

答：处理智能合约事件的步骤：

1. 使用contract.events监听特定事件
2. 实现事件订阅和取消订阅
3. 处理事件数据更新UI
4. 考虑历史事件的获取
5. 处理网络断开重连情况

**DApp前端开发中需要注意哪些安全问题？**

答：主要安全考虑：

1. 私钥安全，避免在前端存储敏感信息
2. 验证智能合约地址
3. 防止重放攻击
4. 交易签名验证
5. 输入数据验证
6. 防止跨站脚本攻击
